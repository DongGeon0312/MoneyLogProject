## 1. 1일차 — 기획·설계·셋업

**목표**: 무엇을 만들지 문서로 확정하고, 빈 프로젝트가 실행되는 상태를 만든다.

| 할 일 | 상태 |
|-------|------|
| 요구사항 정의서 (`docs/requirements.md`) | 완료 |
| ERD 설계 (`docs/erd.md`) | 완료 |
| REST API 명세 (`docs/api-spec.md`) | 완료 |
| 진행 계획 문서 (`docs/progress-plan.md`, 이 문서) | 완료 |
| Spring 프로젝트 셋업 + 엔티티 뼈대 | 완료 (스켈레톤) |
| Git 저장소 생성 + 브랜치 전략 결정 | 진행 중 |
| 로컬에서 프로젝트 실행 확인 (`./gradlew bootRun` 또는 IDE 실행) | 사용자 로컬 환경에서 확인 필요 |

**완료 기준(DoD)**: 요구사항·설계 문서가 저장소에 있고, 앱이 뜨며, 엔티티 뼈대가 컴파일된다.

> Gradle wrapper(gradlew) 생성과 실제 빌드 확인은 인터넷이 제한된 이 작업 환경에서는 수행할 수 없어, 사용자가 로컬(IntelliJ 또는 `gradle wrapper` 명령)에서 진행합니다. 자세한 안내는 대화창 안내 참고.

## 2. 2일차 — 도메인·기본 CRUD

**목표**: 로그인 없이도 도메인이 동작한다.

- 엔티티(User/Category/Transaction)에 JPA 매핑 어노테이션 추가, Repository 완성
- 카테고리 기능: 기본 시드(식비/교통/주거/문화, 급여/용돈) + 추가/수정/삭제 (삭제는 RESTRICT)
- 거래내역 CRUD 구현
- H2에서 저장/조회 동작 확인

**DoD**: H2에서 카테고리·거래내역 생성·조회·수정·삭제 가능, 데이터 실제 저장 확인. (마일스톤: 도메인 동작)

## 3. 3일차 — 인증·인가 (+ 조회·통계는 되는 만큼)

**목표**: "내 데이터만 접근"(인증+인가)이 동작한다.

필수: Spring Security + JWT(BCrypt, accessToken 발급), 본인 데이터 인가(user_id 필터), 기본 입력 검증, 이번 달 합계, Swagger 기본 노출.
선택(여유 시): 카테고리별 지출 집계, 목록 필터(type/categoryId)+페이징, 전역 예외 봉투 통일.

**DoD**: 로그인 후 토큰으로만 보호 API 접근, 남의 데이터는 막힘, 이번 달 합계가 보임. (마일스톤: 인증 동작)

## 4. 4일차 — 프론트 연동 + DevOps 배포

**목표**: 실제로 접속 가능한 서비스가 된다. 오늘의 최우선은 배포.

- 필수 화면 2종(로그인, 거래내역 목록+등록) 구현·API 연동
- Dockerize + docker-compose(앱+MySQL), AWS EC2 배포·실접속 확인
- GitHub Actions CI/CD, (여유 시) 월별 통계 화면 추가

**DoD**: 공개된 EC2 URL로 로그인·거래 등록/조회 가능. (마일스톤: EC2 배포 완료)

## 5. 5일차 — 도전·마무리·발표

**목표**: 결과물을 다듬고 배운 것을 정리한다.

선택한 도전 과제 (기본 완료 후, 우선순위 순): 검색/CSV 내보내기 → 통계 시각화 → Refresh Token → 테스트/커버리지 → Jenkins/K8s 무중단 배포.

- README 정리(실행법·배포 URL·아키텍처), 회고·자기평가 작성
- 제출물 5종 최종 점검: GitHub 저장소, 배포 URL, Swagger 문서, 요구사항·설계 문서, 회고

**DoD**: 제출물 5종 완비. (마일스톤: 발표)

## 6. Git 브랜치·커밋 전략

- `main`은 항상 "동작하는" 상태로 유지
- 기능 단위 `feature/*` 브랜치에서 작업 후 `main`으로 병합
- 브랜치 예시: `feature/auth-jwt`, `feature/category-crud`, `feature/transaction-crud`, `feature/monthly-stats`, `feature/frontend-integration`, `feature/docker-deploy`
- 커밋 메시지: `타입: 요약` (예: `feat: 거래내역 등록 API 구현`, `fix: 금액 검증 조건 수정`)
  - 타입: `feat`(기능), `fix`(버그), `docs`(문서), `refactor`(리팩터링), `test`(테스트), `chore`(설정/빌드)
- 매 일차 마무리에는 반드시 원격 저장소에 push
