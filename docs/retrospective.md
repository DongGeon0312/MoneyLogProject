# 회고 (초안)

이 문서는 초안입니다. 실제로 느끼신 부분, 어려웠던 지점, 다르게 하고 싶은 부분을 자유롭게 고쳐서
동건님 자신의 언어로 완성하시면 됩니다. 아래는 이번 프로젝트 진행 과정에서 실제로 있었던 일들을
정리한 뼈대입니다.

## 1. 무엇을 만들었나
머니로그(MoneyLog) — 개인 가계부 웹 서비스. 회원가입/로그인(JWT), 카테고리·거래내역 CRUD, 월별
통계, 검색/CSV 내보내기, Refresh Token, 정적 프론트(로그인/목록+등록/통계) 화면까지 기본 과제와
선택한 도전 과제(예산 알림 제외, 나머지 전부)를 포함해 구현했습니다.

## 2. 배운 점 (초안 — 자신의 말로 다시 써보세요)
- 요구사항 → ERD → API 명세 → 구현 순서로 진행하니, 코드를 짜기 전에 이미 "무엇을 만들지"가
  명확해져서 구현 중 흔들림이 줄었다.
- 인가(Authorization)는 결국 "모든 조회 쿼리에 user_id 조건을 강제로 건다"는 단순한 원칙이지만,
  이걸 처음부터 설계(ERD의 FK 구조)에 반영해두지 않으면 나중에 손대기 어렵다는 걸 체감했다.
- Spring Security는 의존성을 추가하는 순간부터 기본 동작(전체 인증 요구)이 켜지기 때문에,
  JWT를 완성하기 전 임시로 permitAll을 열어두는 등 "단계적으로 잠그는" 접근이 실무적으로 자연스럽다.
- 공통 응답 봉투(ApiResponse)와 표준 에러 코드를 초반에 정해두니, 이후 컨트롤러/프론트 코드가
  전부 같은 패턴을 따르게 되어 일관성을 유지하기 쉬웠다.

## 3. 어려웠던 점 (초안)
- (예시) 로컬 개발 환경에서 Git/빌드 도구 설정 과정에서 예상치 못한 문제를 겪었다 — 어떤 부분이었는지,
  어떻게 해결했는지 적어보세요.
- (예시) JPA 엔티티에 @Entity 매핑을 빠뜨렸을 때 "Not a managed type" 에러를 처음 봤을 때 원인을
  찾는 과정.
- (예시) 선택 필터(기간/타입/카테고리)가 있는 목록 조회를 파생 쿼리 메서드로 표현하려다,
  "값이 없으면 조건을 무시"하는 게 안 된다는 걸 깨닫고 JPQL로 바꾼 경험.

## 4. 다음에 개선하고 싶은 점 (초안)
- 통계 화면에 월별 추이(라인차트)를 추가하면 카테고리별 지출 외에 시간 흐름도 볼 수 있을 것 같다.
- Refresh Token을 프론트에서 자동으로 사용해, accessToken이 만료돼도 로그인이 끊기지 않게 만들고 싶다.
- 테스트를 서비스 계층 단위 테스트 위주로만 작성했는데, 컨트롤러 통합 테스트(MockMvc)까지 넓히고 싶다.
- `ddl-auto: update` 대신 Flyway로 스키마를 버전 관리하면 더 안전할 것 같다.

## 5. 자기평가 체크리스트
- [ ] 기본 요구사항(F-01~F-10)이 전부 동작한다
- [ ] EC2에 실제로 배포되어 접속 가능한 URL이 있다
- [ ] Swagger UI로 API 문서를 확인할 수 있다
- [ ] 요구사항/ERD/API 명세 문서가 저장소에 있다
- [ ] 커밋 이력이 기능 단위로 의미 있게 쌓여 있다
- [ ] 이 회고를 자신의 경험을 반영해 다시 썼다
